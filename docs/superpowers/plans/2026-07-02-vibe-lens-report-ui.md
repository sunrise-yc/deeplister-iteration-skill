## 已确认但尚未实现的 UI 能力

### 1. 中英文报告界面

- 顶部要有语言切换。
- 界面标签、板块标题、空状态、悬浮框说明要能切换语言。
- 用户写进记录里的内容保持原文，不强行翻译。

### 2. 证据看板布局

页面顺序：

- 顶部栏：产品名、定位短句、生成时间、语言切换。
- 总览数据：当前问题、历史问题、变更文件、代码变化。
- 图例：说明颜色和符号。
- 当前问题 / 任务。
- Code Diff / 代码差异。
- 迭代方向。
- 证据链 / 冲突线索。
- 历史问题。

### 3. 代码差异圆环

- 大圆环展示全项目 Git diff。
- 小圆环展示某个问题关联的代码差异。
- 绿色表示新增行。
- 红色表示删除行。
- 灰色表示暂无明确关联。
- 圆环只展示代码变化量，不代表优先级、风险或价值。

### 4. 证据完整度条

- 中文名称：证据完整度条。
- 英文名称：Evidence meter。
- 放在问题条目文字下方。
- 表示复盘材料是否充分，不表示任务完成度。
- 不使用斜纹、闪烁或装饰纹理。

### 5. 悬浮框

- 数字区只放数字、比例和简短单位。
- 百分比颜色与对应主数字一致。
- 解释区保留自然语言说明。
- 悬浮框不能跑出界面。
- 靠左元素向右展开，靠右元素向左收。
- 移动端或触屏要能通过点击或焦点打开。

### 6. 大板块标题解释

以下标题需要鼠标悬浮解释：

- 图例 / 鼠标悬浮可解释含义。
- 当前问题 / 任务。
- Code Diff / 代码差异。
- 迭代方向。
- 证据链 / 冲突线索。

## 新增未决设计问题

这些问题必须先经过设计草图确认。

### 问题 A：总览数据卡片是否可展开

当前草图里的总览数据只是静态数字。用户提出：这些图标或卡片是否可以点开，展示对应明细。

待确认方向：

- 当前问题卡片：点击后展示当前问题列表和状态分布。
- 历史问题卡片：点击后展示历史问题表格或主题分组。
- 变更文件卡片：点击后展示文件级 diff 表。
- 代码变化卡片：点击后展示新增 / 删除圆环和 top changed files。

产品边界：

- 可以“展开事实”，不能“给行动建议”。
- 不应把点击展开做成任务管理面板。

### 问题 B：沙盘演示功能

当前报告是“静态复盘看板”，还没有真正的“沙盘演示”。

需要先定义沙盘演示的含义。候选解释：

- 时间回放：沿着迭代记录回看项目状态如何变化。
- 情境查看：点击某个问题，看到它关联的证据、代码差异、讨论路径。
- 路径对比：展示原计划、实际推进、放弃路径之间的差异。

产品边界：

- 第一阶段可以做“回放和展示”。
- 不做“模拟未来怎么安排任务”。
- 不做“自动推演最优路径”。

### 问题 C：非线性迭代路径

当前草图把迭代方向做成线性时间轴，但真实项目可能出现：

- 中途换方向。
- 一条支线暂停。
- 两条支线合并。
- 回滚到旧方向。
- 某个方向只是探索，后来放弃。

更合适的展示方式可能是“迭代路径图”，而不是单线时间轴。

候选表达：

- 主线 + 支线图：保留主路径，同时显示探索支线。
- 节点图：每个迭代节点有状态，例如继续、暂停、放弃、合并、回滚。
- 泳道图：按主题分道，例如 UI、Skill、报告、发布、兼容性。

产品边界：

- 图只是展示路径变化，不评价哪条路径更优。
- 如果记录里没有结构化父子关系，就不能硬画分支；只能退回时间线。
