# Vibe Lens 0.2 可视化认知层设计

日期：2026-07-08

## 1. 背景

Vibe Lens 已经完成第一阶段封装：它可以读取 `docs/iteration-record.md`，结合 Git diff 生成 HTML 复盘沙盘，展示当前问题、历史问题、代码差异、证据链、冲突线索和迭代路径。

这次迭代的目标不是新增一个叫“理解 Coding”的独立卡片。用户已明确纠正：

- 整个 Lens 都是用来帮助理解 coding 的。
- 理解 Loop 只是其中一个场景，不是根本定位。
- 产品不应该展示 Agent 的认知模型。
- 产品应该展示足够多的项目事实，让操作者形成自己的项目认知模型。

大白话：Vibe Lens 不是把 AI 的脑子打开给人看，而是把“发生了什么、为什么发生、证据在哪里、哪里还不清楚”摆清楚，让人自己重新看懂项目。

## 2. 一句话定位

Vibe Lens 用一套可视化语言，把 vibe coding 过程中散落的问题、代码变化、证据、路径和风险组织起来，帮助人形成自己的项目认知模型。

## 3. 核心边界

本阶段继续遵守第一阶段边界：

- 不自动排序任务。
- 不自动安排下一步。
- 不给问题加权重。
- 不把 Agent 判断伪装成事实。
- 不要求用户理解每一行代码。
- 不做完整交互平台。

本阶段新增的边界：

- 不新增“理解 Coding”独立卡片，因为这会弱化其他卡片的共同职责。
- 不展示“Agent 怎么想”，只展示“项目事实足不足够让人形成理解”。
- 所有可视化指标都叫信息信号，不叫质量评分、优先级评分或任务完成度。

## 4. 什么是“问题有证据”

“问题有证据”不是说这个问题一定完全正确，也不是说这个问题很重要。

它的意思是：这个问题不是凭空出现的，后面有可以回头核对的依据。

一个问题的证据可以来自：

- 用户明确提出：例如“这个页面跳转不对”。
- Agent 观察到的现象：例如“测试失败，错误信息指向某个函数”。
- Git diff：例如“这轮修改确实碰到了这些文件”。
- 测试或运行结果：例如“单元测试通过/失败，浏览器打开后看到某个现象”。
- 截图、报告、日志或外部文档。
- 迭代记录里的历史决策。

大白话例子：

| 问题 | 没有证据的写法 | 有证据的写法 |
|---|---|---|
| 页面可能有问题 | 页面体验不好 | 用户指出四个总览指标都跳到同一个详情页；关联文件是 `report_template.html`；测试覆盖了详情入口 |
| 代码可能太乱 | 脚本有点复杂 | `lens_snapshot.py` 同时负责解析、Git 采集、模板输出；后续如果继续加字段，可能需要拆出数据层 |
| 迭代路径不够清楚 | 路径图还不行 | 当前路径节点是固定视觉演示，未完全来自真实迭代记录；用户要求后续补齐 |

因此，“问题有证据”更准确的产品含义是：**这个问题可追溯、可核对、可复盘**。

## 5. 证据信号的组成

第一版不做复杂评分，只用轻量完整度信号。

一个问题可以从 5 个角度判断信息是否充分：

1. **来源**：谁提出的？用户、Agent、测试、Git、外部资料？
2. **现象**：看到了什么问题？不是只写一个抽象结论。
3. **关联对象**：涉及哪个文件、模块、页面、功能或路径节点？
4. **验证**：有没有跑测试、打开页面、检查截图、编译、人工确认？
5. **不确定性**：还有什么没确认？哪些只是 Agent 判断？

这 5 项可以形成一个“证据完整度条”。它不是质量评分，只是提醒用户：这条记录的信息够不够支撑复盘。

## 6. 全局可视化语言

### 6.1 进度条

进度条用于表达信息完整度，减少长文字解释。

适用位置：

- 当前问题条目。
- 证据链详情。
- 沙盘演示中的问题卡。
- 后续接管风险提示。

建议拆成三个维度：

- `证据`：来源、现象、关联文件是否清楚。
- `验证`：有没有测试、运行结果、截图或人工确认。
- `解释`：有没有说明为什么要改、为什么这样改、还缺什么。

展示规则：

- 进度条只表示信息是否充分。
- 不使用“分数”“排名”“优先级”等词。
- 鼠标悬浮时说明缺了什么，例如“缺少验证记录”。

### 6.2 圆环

圆环用于表达比例关系。

当前已有的大圆环表示代码新增/删除比例，这个方向保留。

后续可以增加语义：

- 大圆环：新增行 / 删除行。
- 小圆环：某个问题关联的代码变化比例。
- 次级圆环或环形标签：已解释变化 / 未解释变化，已验证变化 / 未验证变化。

限制：

- 如果问题和文件没有明确关联，不能为了圆环好看而编造关系。
- 没有关联时显示“关联不足”或灰色状态。

### 6.3 路径图

路径图用于表达项目为什么走到今天。

当前路径图更像视觉提示，后续要变成数据驱动的认知地图。

节点语义：

- 普通圆点：一次重要迭代。
- 分叉节点：方向选择。
- 灰色节点：放弃路径。
- 绿色节点：已经验证。
- 黄色节点：证据不足或解释不足。
- 红色节点：冲突线索、接管风险或高不确定性。

节点点击后展示：

- 当时的问题是什么。
- 为什么转向。
- 改了哪些文件或模块。
- 有哪些证据和验证。
- 留下了什么认知缺口。

### 6.4 轻量标签

标签用于区分事实来源，避免一大段文字混在一起。

建议固定标签：

- `Git 事实`
- `记录事实`
- `测试验证`
- `操作者确认`
- `Agent 判断`
- `缺证据`
- `缺验证`
- `解释不足`

规则：

- `Agent 判断` 可以展示，但必须和事实分开。
- `缺证据`、`缺验证` 是提醒，不是批评。

## 7. 现有卡片如何承接

### 7.1 总览数据

现有总览保留：

- 当前问题。
- 历史问题。
- 变更文件。
- 代码变化。

每个数字下面增加轻量视觉信号：

- 当前问题：多少问题有证据。
- 历史问题：resolved / open / discussing 的分段条。
- 变更文件：变化是否集中在少数文件。
- 代码变化：新增/删除比例，以及解释/验证状态入口。

目标：总览不只是“数量面板”，而是“项目理解状态的入口”。

### 7.2 当前问题

每个问题条目保留标题、状态、证据和关联文件。

新增三条短信号：

- 证据完整度。
- 验证完整度。
- 解释完整度。

如果空间不足，主页只展示一条总的“信息完整度”，详情页再拆三条。

### 7.3 代码差异

代码差异卡继续使用圆环展示新增/删除比例。

文件列表新增状态标记：

- `已解释`
- `未解释`
- `有验证`
- `缺验证`

这些标记来自记录文件中明确的文件关联和验证记录。没有明确数据时，显示灰色的“未关联”，不猜。

### 7.4 证据链 / 冲突线索

这块从文本总结升级为“事实类型面板”。

展示方式：

- 左侧展示证据来源标签数量。
- 右侧展示当前缺口：缺证据、缺验证、解释不足、冲突线索。
- 点击进入详情后，按问题或文件分组展示证据。

### 7.5 迭代路径

路径图要从固定演示图升级为真实迭代记录生成。

第一版可以只解析 `## 迭代记录` 下的标题，生成顺序节点。

后续再支持：

- 分叉。
- 放弃路径。
- 验证节点。
- 风险节点。
- 节点展开。

### 7.6 沙盘演示

沙盘演示页可以变成“事实拼图”：

- 左侧：问题。
- 中间：相关文件和代码变化。
- 右侧：证据、验证和缺口。
- 底部：它在迭代路径里的位置。

目标：让用户不用读完全部 Markdown，也能看懂一次 coding 为什么发生、碰了哪里、还有什么没解释清楚。

## 8. 数据模型建议

不建议第一版引入复杂数据库。继续以 Markdown 作为数据源，脚本输出 JSON。

可以在记录格式里逐步支持这些字段：

### 8.1 问题池字段

当前已有：

```md
| 编号 | 问题 | 来源 | 状态 | 证据 | 关联文件 |
```

后续可以扩展为：

```md
| 编号 | 问题 | 来源 | 状态 | 证据 | 关联文件 | 验证 | 解释状态 |
```

字段含义：

- `验证`：测试、截图、人工确认、运行命令。
- `解释状态`：已解释、部分解释、解释不足。

### 8.2 迭代记录模板

每次重要迭代可以补一段：

```md
认知变化：
- 本轮之后更清楚了什么：
- 仍然不清楚什么：
- 下次接手前应该先看什么：
```

这段不是为了让记录更长，而是为了减少未来接手时的断片。

### 8.3 可选新章节

如果问题池字段变得太长，再新增：

```md
## 认知缺口记录
```

但第一版不优先新增章节，先复用问题池和迭代记录，避免记录文件变重。

## 9. 第一版实施范围建议

第一版只做三件事：

1. **补强进度条语义**
   - 把现有“证据完整度”扩展为证据、验证、解释三个概念。
   - 主页紧凑展示，详情页展开说明。

2. **补强代码差异卡**
   - 保留新增/删除圆环。
   - 文件列表增加解释/验证状态标记。
   - 没有数据时显示“未关联”，不猜。

3. **补强迭代路径图**
   - 路径节点来自真实 `## 迭代记录` 标题。
   - 节点显示验证/缺口/风险的轻量标记。
   - 复杂分叉和拖拽留到后续。

这三件事正好对应用户提出的进度条、圆环、迭代路径图，也能显著降低信息负担。

## 10. 测试与验证

需要增加或更新测试：

- Snapshot 能读取新增的 `验证`、`解释状态` 字段。
- 没有这些字段时，旧记录仍能正常显示。
- HTML 报告不出现“理解 Coding 独立卡片”。
- HTML 报告保留当前问题、代码差异、证据链、迭代路径等现有入口。
- 证据完整度文案明确说明它是信息完整度，不是优先级或质量评分。
- 文件 diff 标记只有在有明确关联数据时才显示已解释/有验证。
- 迭代路径节点来自真实迭代标题，而不是只用固定演示节点。
- 生成 HTML 后语言切换、详情页跳转、设置面板仍然可用。

手动检查：

- 主页没有因为新增进度条和标签变得更乱。
- 小屏幕下标签不会挤压问题标题。
- 鼠标悬浮解释不会溢出页面。
- 用户能一眼区分事实、Agent 判断、缺证据和缺验证。

## 11. 不在本次范围

本次不做：

- 完整交互平台。
- 拖拽整理问题。
- 自动生成复杂关系图谱。
- 自动判断任务优先级。
- 自动推荐下一步任务。
- 自动猜测每个文件为什么被改。
- 展示 Agent 内部认知模型。
- 复杂 Loop 回放。

## 12. 验收标准

本阶段完成后，用户应该能从报告里更快回答：

- 当前哪些问题有证据，哪些只是被记录了标题。
- 这轮代码变化哪些能解释，哪些还不能解释。
- 哪些改动验证过，哪些缺验证。
- 项目最近几轮是怎样走到现在的。
- 如果下次继续做，哪些地方最需要先看清楚。

同时，报告不能给用户一种错觉：Lens 已经替人判断好了下一步应该做什么。
